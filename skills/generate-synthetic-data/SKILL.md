## What this skill is

The research agent's **execution-regime synthetic-data stage**: when the **Stage-14 methodology** calls for synthetic or simulated data as the thing the study actually analyzes, this skill generates it, validates it against a real benchmark, and stages it — with a written record of *how it was generated* and *why it can be trusted.* It is the execution sibling of `source-and-prepare-data`, and the **synthetic-as-object** third leg of the three-way distinction:

- `operationalize-construct` (planning) — may build a **toy/synthetic exploration surface** for *trying* operationalizations only; **never analyzed, never validates** a measure, never enters the paper.
- `source-and-prepare-data` (execution) — acquires **real data only**; never generates analyzed data; if no real dataset is reachable it routes to the research teacher or faculty mentor (gated access), **never** substitutes synthetic.
- **`generate-synthetic-data` (execution, this skill)** — generates synthetic/simulated data **as the legitimate studied object**, only when the **Stage-14 methodology** specifies it, validated by **algorithmic fidelity** against a real benchmark, labeled synthetic wherever written.

**The dividing line between the toy surface and this skill is the *method*.** Synthetic-as-object must be the **chosen, sealed design** — not a workaround reached for because real data is hard to get. If synthesis is being considered *because* the real dataset is unreachable, that is not this skill: it is `source-and-prepare-data`'s gated-data route (record the block, route to the research teacher/faculty mentor). This skill runs only on a design that *chose* synthesis on the merits and froze that choice at the seal.

By completion the student has:

- A **labeled synthetic dataset** staged in the workspace `data/` folder (clearly named synthetic), plus the **generator itself** — the script / prompt set / agent configuration, with the model version and seeds pinned so the run reproduces.
- The **Dataset Creation Log** (synthetic variant, `data/{core|extension}-dataset-creation-log.md`) — the running record of all seven steps (plan, teaching, the Review Agent's plan review, the plan approval, the execution report, the Quality Control (QC) audit, the spot-check results), absorbing the old generation-and-fidelity fields: the generation spec (model + version, personas/prompts/parameters, seeds, provider, date); the **fidelity benchmark and its result**; and the explicit "labeled synthetic" declaration — the trail the **Dataset Creation Paragraph** and the Review Agent both draw on.
- A clear **fidelity verdict**: the synthetic data reproduced a known real benchmark for the checkable cases (trustworthy for the novel cases), or it did not (not analysis-ready — revisit the generator, or **post-seal** refine against what was found and keep the relevant paragraph current, describing it in the Dataset Creation Paragraph — rigorous practice).

It never presents synthetic data as empirical, never lets a synthetic set substitute for unreachable real data, and labels synthetic data **synthetic** wherever it is written — in `data/`, in the record, in the Methods, and in every figure and table.

## Reconciling the SOUL's "I never invent data"

The research-agent SOUL commits: *"I never invent data, and I never invent a source."* That rule is about never fabricating observations to stand in for reality or to manufacture findings. It is **not** contradicted here — the SOUL itself names "synthetic-participant runs" as a legitimate data mode. Generating synthetic data is legitimate, not fabrication, when **all three** hold:

1. **Sealed design, not convenience.** Synthesis is the methodology the student *chose and sealed*, on the merits — never a fallback for data they could not obtain.
2. **Fidelity-validated.** The synthetic data reproduces a known real benchmark for the cases that can be checked, before its novel outputs are trusted. Un-validated synthetic data certifies nothing (the direct analogue of `operationalize-construct`'s "a toy set never yields a `validated` verdict").
3. **Labeled synthetic everywhere.** Nothing generated is ever presented as an empirical observation.

Strip any one of the three and it collapses back into "inventing data" — which is exactly what the SOUL forbids. This skill's whole job is to hold all three.

## When this skill fires

In the research agent's **execution regime**, after the proposal is **sealed** (Stage 16 — the proposal is committed, including the chosen synthetic design and the **Hypotheses and/or Expected Findings**; the methods themselves keep evolving during execution as the student learns) and the one project-level IRB (Stage 17) is navigated, at a **stream's Data stage** and before that stream's analysis runs. Execution bifurcates **core-then-extension**, so this skill supplies whichever stream's Stage-14 methodology specifies synthetic-as-object: the **core** dataset at **Stage 18** (handing forward to the core Analysis, **Stage 19**) or the **extension** dataset at **Stage 22** (handing forward to the extension Analysis, **Stage 23**). If both streams are synthetic-as-object it fires once per stream; if only one is, it fires for that stream while `source-and-prepare-data` serves the other's real data. Two entry paths:

- **Seal-dispatched** — the mentor composes the concrete handoff into execution (the R1/R10 pattern: the concrete switch to the research agent — icon-click on desktop; `research` on the TUI — plus a prompt naming the sealed synthetic methodology). The research agent recognizes the arriving handoff per R1/R9 and reads the named files first.
- **Ceded-to from `source-and-prepare-data`** — when that skill hits a **shape-E** design (or a declared cross-shape synthetic role), it cedes here rather than generating analyzed data itself.

**Why generation waits for the seal (deliberate, not an oversight).** Same logic as `source-and-prepare-data`: planning (Stage 5) validates the *operationalization* — for a synthetic design, whether the persona/probe/agent-rule can in principle carry the construct — on cheap, revisable ground. Standing up the full generator and its fidelity benchmark before the seal would be **wasted effort** if the design still changes (a failed planning check bounces back to revise the research question; the §7 stale map cascades). The seal is the commitment point that makes the one-time heavy generation-and-validation work worth doing. Division: **validation of the idea (planning, cheap, revisable)** versus **generation of the object (execution, post-seal, committed)**.

## Dispatch and the handoff (the execution-regime shape)

Like `source-and-prepare-data`, this skill lives in **execution**, the research agent's own regime — it does not return to the mentor on clean completion.

- **Inbound** — seal-dispatched (R1/R10) or ceded-to from `source-and-prepare-data` (shape E). The agent reads the Stage-14 methodology + the Stage-5 operationalization record first.
- **Forward, not back.** On a clean completion this skill hands **forward to analysis** (`conduct-analysis`, Tier 6), where the agent executes the analysis code (permitted tooling) producing the **Data Analysis Log** (`analysis/{core|extension}-data-analysis-log.md`), preconditioned on the validated synthetic dataset + the Dataset Creation Log. Execution stays the research agent's lane.
- **Two Review-Agent checkpoints, scheduled in the plan (steps 3 and 7).** The plan schedules both, and **each produces a dated review `.md` in the workspace `reviews/` folder**. For the **step-3 plan review** (pre-execution) the research agent composes a full handoff prompt — pointing the **Review Agent** at the relevant docs and instructing it to write its review to a dated file in `reviews/`; the student carries the prompt across, and the research agent later **retrieves** that dated file, revises the plan, and re-teaches. For the **step-7 QC audit** (post-execution) the research agent again composes the handoff prompt (Review Agent pointed at the report + the executed output, instructed to write a dated audit file in `reviews/`); the audit flags any unexpected findings and suggests **3–5 spot checks the student runs** to catch AI error (V5 function 2), and the student reports the results back. The step-7 audit covers the surface most exposed to agent error: did the generator run the sealed spec, is the model version pinned and recorded, are the outputs non-degenerate (no duplicated rows / mode-collapse / a persona that ignored its conditioning), does the N match; and does the fidelity benchmark actually calibrate this data rather than being cherry-picked, is the fidelity claim precise about what it did and did not check, are the un-checkable novel outputs being trusted no further than the benchmark warrants. The existing R11 spot-check points **are** that step-7 QC-audit substance. Routing these to the dedicated guard (rather than self-checking) is the point.
- **R8 (position file) — candidate (c).** This skill does **not** write `project_paper_status.md`. Execution progress lives in the execution-track records it *does* own (`data/`, `data/{core|extension}-dataset-creation-log.md`, `decisions.md`); the paper/position stage advances **mentor-side** when the student returns for the writing handoff (`project-walkthrough` receives the return and advances the stage; `paper-walkthrough` dispatches `scaffold-section`). The research agent signals via the student's return, never by writing the position file (2026-07-01 ruling; aligns with V2/M1).
- **The fidelity-failure loop-back.** If the synthetic data cannot reproduce the benchmark, the handoff loops **back** — to `operationalize-construct` (if a planning revision is still open) or, **post-seal**, the shortfall is refined against: the methods evolve as the student learns, and the student keeps the relevant paragraph current, describing what they actually did in the Dataset Creation Paragraph (with the trail in the Log). The one thing frozen is the **Hypotheses and/or Expected Findings**.

## What "generate" means depends on the method — the menu-shape layer

The three-phase spine below is **menu-agnostic** — specify the generator, generate + label, validate by fidelity — but what "generate" concretely produces depends on the Stage-14 method. This skill spans two use-families:

### Family 1 — Synthetic as the *object* (shape E)

The findings come *from* the synthetic data; validity is algorithmic fidelity against a real benchmark (mirrors `operationalize-construct` shape E).

- **E1 — Silicon samples / synthetic AI survey participants.** *Persona-conditioned LLM responses as respondent data (Argyle "silicon sampling"; Horton "homo silicus").* Generate: a persona/conditioning scheme + the elicited measure, one simulated respondent at a time. Fidelity: reproduce known real marginals/subgroup patterns from a real survey (ANES, GSS) for the checkable cells before trusting the novel ones. *(Sketch; per-exemplar.)*
- **E2 — Generative agent-based simulation.** *Interacting agents producing emergent social dynamics (Park generative agents; OASIS / MiroFish).* Generate: agent rules + a generative push + the emergent outcome measure. Fidelity: reproduce a known stylized fact / real dynamic the system should recover, before reading off the counterfactual runs. *(Sketch; per-exemplar.)*
- **E3 — Fine-tuned-model probe.** *Fine-tune a model on a community corpus, then probe it (Jiang CommunityLM).* Generate: the fine-tune + the probe protocol. Fidelity: validate the probe against a real benchmark for that community (CommunityLM probed against ANES). *(Straddles shape D on the surface — the probe looks D-like — but the validity step is E's: calibrate against a real benchmark. Sketch; per-exemplar.)*

### Family 2 — Synthetic as a *subordinate auxiliary* (cross-shape roles)

The **real data is still the object**; the synthetic data plays a declared, sealed supporting role. Labeled synthetic and fidelity-checked all the same, but it never becomes the finding.

- **Privacy-preserving synthesis.** Generate a synthetic twin of a sensitive real dataset so the analysis is shareable. Fidelity: the synthetic preserves the real distributions and key relationships (utility) without re-identifying anyone. *(Sketch.)*
- **ML augmentation.** Synthesize additional training rows for a classifier/measurement model. Fidelity: held-out performance gain **on real data**, no train/test leakage. *(Sketch.)*
- **Monte-Carlo / bootstrap for inference stability.** Resample or simulate to obtain CIs, permutation nulls, or a sampling distribution. This simulates *the estimator*, not fabricated observations. Fidelity: the resampling scheme is correct and reported as such. *(Sketch.)*
- **Simulation for power / robustness.** Simulate data under pre-specified assumed effects to compute power or stress-test the design. Fidelity: the generating model matches the pre-registered assumptions; results are reported as **simulation**, never as empirical findings. *(Sketch.)*

**Build-as-you-go flag (load-bearing, mirrors the two sibling skills).** There is **no gold exemplar in this skill yet**, so **every use-form above is a sketch.** Each is fleshed out to full detail when a gold exemplar in that use is built. Until a use has an exemplar, a project in it runs the spine using the sketch **plus a declared deferred-scope note** in `data/{core|extension}-dataset-creation-log.md` — so the gap is explicit, never latent. (Precedent: `source-and-prepare-data` shape A specified / B–F sketched; `operationalize-construct` the same.)

## The seven-step loop (plan → teach → execute → report)

Whichever stream is synthetic-as-object (a core at Stage 18 or an extension at Stage 22) runs the same seven-step loop from the execution-plan model (`design/execution-plan-model-2026-07-07.md`). The gate is at the **plan** level — teach the plan, review it, confirm understanding, approve — not per line of code. AI running generation and fidelity code is permitted research tooling under STS (disclosed and cited); the agent writes and runs it directly. The student directs, understands, and owns the work.

**Step 1 — Plan.** The agent writes an execution plan whose purpose is to **teach**: the principles behind synthetic generation and fidelity validation for this shape, the ordered steps, and the **specific scripts it will write and run** (the generator script / prompt-set / agent config, the fidelity/benchmark script — named in the plan, with model version and seeds pinned). It marks any **human-in-the-loop step** it cannot do itself as *"you have to do this for me — here are the steps"* (e.g., a credentialed model/provider account, or — for privacy-preserving synthesis — the gated access to the sensitive real source, which is `source-and-prepare-data`'s route). The plan also **schedules its two Review-Agent checkpoints** — the step-3 plan review and the step-7 QC audit, each writing a **dated review `.md` to the workspace `reviews/` folder**.

**Step 2 — Teach.** The agent teaches the plan and answers questions until the student **signals understanding**. Teaching lives up-front, in the plan.

**Step 3 — Review-Agent plan review.** On the understanding-signal, the agent **composes a full handoff prompt** — pointing the **Review Agent** at the relevant docs and instructing it to write its plan review to a **dated `.md` in `reviews/`**. The student carries the prompt across; the research agent later **retrieves** the dated file, processes the changes, and **loops back through steps 1–2** (revise + re-teach) as needed.

**Step 4 — Approve.** The reviewed, revised plan is approved **jointly by the student (understanding) and the teacher**, before anything is generated.

**Step 5 — Execute (the three phases below).** The agent **writes and runs the code directly** — specify, generate + label, validate by fidelity — following the approved plan; individual coding acts are not gated. The three phases are the concrete content of this step.

**Step 6 — Report.** The agent writes a **post-execution report** into the Dataset Creation Log: what it generated, the fidelity result and diagnostics, and **any revision to the plan** (a spec that changed, a place where the method evolved as the fidelity work was learned — the student keeps the relevant paragraph current, which is rigorous practice).

**Step 7 — Review-Agent QC audit + spot-checks.** The agent **composes a second handoff prompt** — pointing the **Review Agent** at the **report + the generated output** and instructing it to write a **dated QC-audit `.md` to `reviews/`**; the audit flags any unexpected findings and suggests **3–5 spot checks the student runs** to catch any hallucination in the generation/validation step (degenerate outputs, an over-claimed fidelity match). The research agent **retrieves** the dated audit file; the student runs the spot checks and reports back, and the agent prompts for this — *"tell me what you found so I can add it to your log."* **Loop-back:** an error that compromises the whole dataset → full re-run + re-report + re-audit (not approve-complete until the audit comes back clean); a smaller error → fix + a new report. Everything is logged.

## Step 5 in detail — the phases (specify, generate, validate)

### Phase 1 — Specify the generator (from the Stage-14 methodology)
Pin down exactly what is being generated and how, drawn from the Stage-14 methodology and the Stage-5 operationalization record — the persona/conditioning + elicited measure (E1), the agent rules + emergent outcome (E2), the fine-tune corpus + probe (E3), or the declared cross-shape role (Family 2). Record the **generation spec** in `data/{core|extension}-dataset-creation-log.md`: model + exact version, prompts/personas/parameters, random seeds, provider, and date (an LLM's outputs drift with model version, so the version is part of provenance, not a footnote). Name the **real benchmark** the fidelity check (Phase 3) will use — chosen *now*, before generating, so it is not reverse-engineered to pass.

**The bright-line gate (internal — not a student question).** Before generating anything, confirm this is the **sealed synthetic design**, not a reach-around for unreachable real data. This is an **internal routing check keyed on the seal and the dispatch, not a conversation to have with the student** — by the time this skill fires the design is sealed, so synthesis is already the chosen approach; the check simply guards the one pathological case where synthesis is being reached for *because* real data turned out unreachable. In that case STOP: that is `source-and-prepare-data`'s gated-data route (record the block; route to the research teacher or faculty mentor), never substituted here. A **cede-in from `source-and-prepare-data`** is, by construction, a genuine **shape-E chosen-design** cede — never a gated-data reroute (that sibling's unreachable-real-data case exits to the research teacher, and never here) — so this guard's pathological case can only arise from a **mis-sealed design**, not from the sibling's handoff. *(An earlier draft surfaced this as a scripted student question; removed 2026-07-01 on educator ruling — the seal already settles whether synthesis is the design, so asking would re-open a closed question. The guard survives as this internal rule plus the "Synthesis reached for as a workaround" edge case.)*

### Phase 2 — Generate + label
Run the generator to produce the synthetic dataset/run, **labeled synthetic wherever it is written** (file names, the record, later the Methods and every figure). Keep the full **generation log** — model version, seeds, parameters, date — so the run reproduces; a synthetic dataset whose generator cannot be re-run is not a defensible object. Stage the labeled dataset (and the generator script/config) in `data/`. Major generation choices also go to `decisions.md` (five-field).

→ **Feeds the step-7 QC audit (generation).** The Review Agent's audit covers: did the generator run the sealed spec, is the version pinned, are the outputs non-degenerate, does the N match — spot checks the student runs on the executed output.

### Phase 3 — Validate by algorithmic fidelity (the integrity gate)
Before the synthetic data is declared analysis-ready: **reproduce the pre-named real benchmark** for the cases you can check, to earn trust in the novel outputs you cannot. The check is use-specific — silicon-sample marginals vs ANES/GSS; a generative simulation recovering a known stylized fact; a fine-tuned probe vs a real survey; a privacy-twin's distributions/correlations vs its real source; augmentation's held-out gain on real data; MC/bootstrap coverage; power-sim calibration. Record the benchmark, the comparison, and the outcome.

A fidelity **failure** = the synthetic data is not trustworthy: **do not analyze it.** Either the generator needs revisiting (a fixable spec problem) or the design's synthetic premise does not hold up (a design-level problem). Telling those apart is the load-bearing call, so at the verdict the skill surfaces the research agent's **careful-tier** option (the SOUL's "How I run, and when to use a stronger model": suggest switching to the careful tier for this one decision, then dropping back). It does **not** reference the mentor's `model-escalation` meta-skill — that ships only in the project-mentor profile; the research agent's escalation surface is its own SOUL, which keeps the tiers model-agnostic. Fire it once, at the verdict — not per check.

> **Scripted moment — the fidelity gate** *(voice-read approved 2026-07-01; post-seal clause reworded + data-paragraph naming refreshed 2026-07-07 — pending re-read).* Intent: no findings off un-calibrated synthetic data; slow down for the load-bearing version. *Agent:* "Here's the moment that decides whether any of this is usable. We agreed up front on a real benchmark this synthetic data should be able to reproduce — a number or a pattern we already know is true out in the world. If it hits that, we've earned some trust in the parts we *can't* check directly, and we can go on to analyze. If it misses, that's not a small thing: it means the generated data isn't behaving like reality, and nothing we'd find in it would be real. Two possibilities, and they're very different — one, the generator has a fixable bug and we re-run it; two, the whole synthetic premise doesn't hold for this question — and your methods can evolve as we learn; the one thing that stays frozen is your Hypotheses or Expected Findings. So we refine against what we found and describe what we actually did in your Dataset Creation Paragraph, which is exactly the rigorous thing to do. Telling those apart is exactly the kind of call worth switching to the careful tier for. Show me the benchmark: what did we expect it to reproduce, and what did it actually produce?"

*(Scope note — Family 1 vs Family 2: the moment's "nothing we'd find in it would be real" framing is written for the **synthetic-as-object** case (Family 1), where the finding comes from the synthetic data. For a **Family-2 auxiliary** use (MC/bootstrap, power-sim, augmentation, privacy-synthesis) the real data is still the object, so a fidelity miss means the synthetic **aid is discarded and the real-data finding stands** — not that "nothing is real"; the spoken line is adjusted to that when the use is auxiliary.)*

## What this skill writes to the workspace
- **`data/`** — the **labeled synthetic dataset** and the **generator** (script / prompt set / agent config, with seeds), staged locally; large generated artifacts are git-ignored like `articles/` (local-by-default), with the generator script kept so the data reproduces.
- **`data/{core|extension}-dataset-creation-log.md`** — the **Dataset Creation Log** (synthetic variant): the whole seven-step trail (plan, teaching notes, the Review Agent's plan review, the plan approval, the execution + report, the QC audit, the spot-check results and what the student found), absorbing the old generation spec (model + version, personas/prompts/params, seeds, provider, date), the fidelity benchmark + result, and the "labeled synthetic" declaration. The agent prompts the student to report back what the review, the audit, and the spot checks turned up — *"tell me what you found so I can add it to your log."* This is the durable execution-track record — the synthetic stream's counterpart to the real-data **Dataset Creation Log** (each stream's Log lives at `data/{core|extension}-dataset-creation-log.md`).
- **`reviews/`** — the **dated Review-Agent files**: the step-3 plan review and the step-7 QC audit each land here as a dated `.md`. The research agent composes the handoff prompt (pointing the Review Agent at the docs and instructing the dated file), the student carries it across, and the research agent later **retrieves** the dated file (the student runs the step-7 spot checks and reports the results back).
- **`decisions.md`** — the five-field generation decisions (generator design, benchmark choice, major parameter/persona choices).
- **does NOT write `project_paper_status.md`** (R8 candidate c). Execution progress lives in the records above; the paper/position stage advances mentor-side at the R10 return.
- **feeds, does not write — the Dataset Creation Paragraph.** The **Dataset Creation Paragraph is the student's** — a summary Methods paragraph narrating the **whole data-stage arc**: the (synthetic) data source, the generation spec, the fidelity check, any amendments, **and the QC steps — including the student's own QC and spot-checks of the agents' work** (a Regeneron contribution strength) — written via an **R10 `scaffold-section` handoff** that draws on the Log (function-not-content: the skill writes the Log, the student writes the paragraph; the Dataset Creation Log stays the ungraded workbench trail that feeds it). Per the 2026-07-01 R10 ruling, the handoff is a **return to the mentor** — the research profile does not carry `scaffold-section`. This graded paragraph is the data stage's paper output; there is no separate execution "Methods Section."

## Cross-agent handoff
Seal-dispatched by the mentor (R1/R10 inbound) or ceded-to from `source-and-prepare-data` on a shape-E design. On clean completion, hands **forward** to analysis (within execution) — the agent executes the analysis in `conduct-analysis`, producing the ungraded **Data Analysis Log** — with the **two Review-Agent checkpoints** (step-3 plan review, step-7 QC audit + 3–5 student spot-checks) scheduled in the plan and an **R10** writing handoff (return to the mentor) for the **Dataset Creation Paragraph**. On a **fidelity failure**, the handoff loops **back** to `operationalize-construct` (planning) or is refined post-seal — the methods evolve as the student learns, and the student keeps the relevant paragraph current in the Dataset Creation Paragraph. If it turns out the real reason for synthesis was unreachable real data, it routes **back to `source-and-prepare-data`'s gated-data path** (research teacher / faculty mentor), never forward. Discharges **R7**'s synthetic-participant-runs clause.

## Edge cases
- **Synthesis reached for as a workaround.** Out of scope by rule — the Phase-1 bright-line gate: if real data is merely unreachable, route to `source-and-prepare-data`'s gated-data path (research teacher / faculty mentor), never synthesize. Synthetic-as-object must be the sealed, chosen design.
- **The synthetic data fails fidelity.** The Phase-3 gate: pre-seal → `operationalize-construct`; post-seal → refine against what was found and keep the relevant paragraph current, describing it in the Dataset Creation Paragraph (with the trail in the Log). The methods evolve as the student learns; the one thing frozen is the **Hypotheses and/or Expected Findings** — never "analyze it anyway."
- **Model-version drift.** LLM outputs change with the model version; the version + seeds are part of provenance, and a re-generation on a new version is a new run, re-validated and re-reported — the version change is recorded, not left implicit.
- **Cross-shape synthetic that drifts into becoming the object.** In Family 2 the real data stays the finding; if an augmentation/privacy/simulation aid starts standing in for the real result, that is a design drift to surface, not proceed through.
- **Un-benchmarkable synthesis.** If no real benchmark exists to calibrate against, the fidelity verdict cannot be earned — the object is `unvalidated` — recorded as such in `data/{core|extension}-dataset-creation-log.md` (the record this skill owns; it never writes `project_paper_status.md`), the shape-E analogue of `operationalize-construct`'s guard: surface it, do not treat novel outputs as trustworthy.
- **Student data / PII.** Same rule as the sibling: synthesizing *from* a sensitive real source (privacy-preserving synthesis) still handles that source under the program privacy rules and the consent-to-publicness IRB line; student-generated PII never enters shared analysis or any model.
