## What this skill is

The research agent's **execution-regime real-data stage**: it takes the **already-validated, already-named dataset** — selected and construct-validated in planning by `operationalize-construct` (Stage 5) and named in the Stage-14 methodology — and converts it into a **real, analysis-ready dataset plus a written record of where it came from and how it was prepared.** The mentor's planning-time reachability check asks *"can we get data for this?"*; `operationalize-construct` answers *"does this dataset actually measure it, and how?"*; this skill *gets it, prepares it, and QCs it.*

It is the execution counterpart of `operationalize-construct`, and the **real-data** sibling of `generate-synthetic-data`:

- `operationalize-construct` (planning) — decides *how each construct is measured and in which dataset, and proves the measurement is sound*; may explore on a **toy/synthetic** surface that is **never analyzed and never validates** anything.
- **`source-and-prepare-data` (execution, this skill)** — **acquires the real dataset**, cleans it, runs Quality Control (QC) on it. It handles **public research datasets only** and **never invents data**.
- `generate-synthetic-data` (execution) — generates synthetic/simulated data **as the legitimate studied object** when the Stage-14 methodology calls for it (shape E, and the cross-shape synthetic uses), validated by algorithmic fidelity. That is a different skill; this one routes there and does not do it.

By completion the student has:

- A **cleaned dataset** staged in the workspace `data/` folder, with the exact subset/variables the analysis needs.
- The **Dataset Creation Log** (`data/{core|extension}-dataset-creation-log.md`) — the running record of all seven steps (plan, teaching, the Review Agent's plan review, the plan approval, the execution report, the QC audit, the spot-check results), absorbing the old provenance/cleaning/QC content: source + access path, every cleaning decision, the analytic N, and the QC checks — the trail the **Dataset Creation Paragraph** and the Review Agent both draw on. It **carries forward** the construct→variable mapping and the construct-validity verdict produced upstream by `operationalize-construct`, rather than re-deriving them.
- Anything the *acquired data* reveals that should **loop back** — e.g., a variable that, on the real file, does not behave as the codebook promised. Post-seal, the **methods evolve as the student learns**: the student refines against the real data and **keeps the relevant paragraph current with what they actually did**, describing it in the Dataset Creation Paragraph. The one thing frozen is the **Hypotheses and/or Expected Findings**; everything else about method and analysis can evolve, and keeping the paragraph current is rigorous practice.

It never invents data, never fabricates values for missing cases, and (per program privacy rules) handles **public research datasets only** — student records are never training or analysis fodder for shared work.

## When this skill fires

In the research agent's **execution regime**, after the proposal is **sealed** (Stage 16 — the proposal is committed, including the named-and-validated dataset and the **Hypotheses and/or Expected Findings**; the methods themselves keep evolving during execution as the student learns) and the one project-level **IRB** (Stage 17) is navigated, then before that stream's analysis runs (core **Stage 19**, extension **Stage 23**, as-executed). It is the **first execution-regime skill**: the cycle seal hands the student out of the mentor-led planning regime into execution, which is the research agent's continuous lane.

It can re-fire if acquisition reveals a measurement problem (below), and it cedes to `generate-synthetic-data` when the Stage-14 methodology is a synthetic/simulation design.

**Runs twice per cycle — once per stream (core-then-extension).** Execution now bifurcates: the student learns the method on the **core**, then applies it to the **extension**. This skill therefore fires **twice**, each time after the single project-level IRB (Stage 17):

- **Stage 18 — Core → Data.** Acquire, clean, and QC the **core** dataset; produce the core stream's cleaned file + the **Core Dataset Creation Log** (`data/{core|extension}-dataset-creation-log.md`) + the graded **Core Dataset Creation Paragraph** (via the R10 writing handoff).
- **Stage 22 — Extension → Data.** Acquire, clean, and QC the **extension** dataset (the same source restricted/extended differently, or a distinct source); produce the extension stream's cleaned file + the **Extension Dataset Creation Log** + the graded **Extension Dataset Creation Paragraph**.

Each run is self-contained: its own dataset, its own provenance record, its own Methods subsection. Where the extension is *within the core's family* the two data records overlap heavily (the extension record points back to the core's rather than re-deriving); where the extension is a distinct family the extension run is fully independent. Each Data stage hands **forward** to that stream's **Analysis** stage — **Stage 19** (core), **Stage 23** (extension) — where the agent executes the analysis code (permitted research tooling under STS, disclosed + cited) in `conduct-analysis`, producing the ungraded **Data Analysis Log** (`analysis/{core|extension}-data-analysis-log.md`), distinct from this data stage and from the later Results write-up.

**Why acquisition and cleaning wait for the seal (deliberate, not an oversight).** Planning (Stage 5, `operationalize-construct`) deliberately validates the operationalization on a **codebook** (and at most a small sample or a labeled toy set), not on a cleaned analytic file — because acquiring and cleaning the full dataset before the seal would be **wasted effort** if the design still changes (a failed validity check bounces back to revise the research question; the §7 stale map cascades). The seal is the commitment point that makes the one-time heavy data work worth doing. So the division is **validation (planning, cheap, revisable)** versus **preparation (execution, post-seal, committed)** — not an arbitrary stage boundary.

This is the load-bearing difference from `operationalize-construct`, which lives in planning and returns the student to the mentor. This skill lives in **execution**, the research agent's own regime.

- **Inbound — seal-dispatched.** At the cycle seal (Stage 16) the **mentor composes the concrete handoff into execution** (the R1/R10 pattern: the concrete switch to the research agent — icon-click on desktop; `research` on the TUI — plus a prompt naming the Stage-14 methodology and the validated dataset from the Stage-5 record). The research agent recognizes the arriving handoff per R1/R9 and reads the named files first. (Both Data runs — core at Stage 18, extension at Stage 22 — sit downstream of this one seal and the one project-level IRB at Stage 17.)
- **Forward, not back.** On a clean completion this skill does **not** return the student to the mentor (planning's pattern). Execution stays the research agent's lane, so it hands **forward to analysis** (`conduct-analysis`, Tier 6 — that stream's Analysis stage: **Stage 19** for the core run, **Stage 23** for the extension run), which preconditions on this skill's prepared dataset + Dataset Creation Log.
- **Two Review-Agent checkpoints, scheduled in the plan (steps 3 and 7).** The plan schedules both, and **each produces a dated review `.md` in the workspace `reviews/` folder**. For the **step-3 plan review** (pre-execution) the research agent composes a full handoff prompt — pointing the **Review Agent** at the relevant docs and instructing it to write its review to a dated file in `reviews/`; the student carries the prompt across, and the research agent later **retrieves** that dated file, revises the plan, and re-teaches. For the **step-7 QC audit** (post-execution) the research agent again composes the handoff prompt (Review Agent pointed at the report + the executed output, instructed to write a dated audit file in `reviews/`); the audit flags any unexpected findings and suggests **3–5 spot checks the student runs** to catch AI error (V5 function 2), and the student reports the results back. The old download-integrity and cleaning sanity checks are the substance the step-7 audit covers — right rows/waves, no skipped records, file N vs the codebook; missing-data handling, recode correctness, the load doing what the log claims — and the existing R11 spot-check points **are** that step-7 QC-audit substance. Routing these to the dedicated guard (rather than self-checking) is the point.
- **The measurement-failure loop-back.** If preparation reveals the data does not carry a construct as validated, the handoff loops **back** — to `operationalize-construct` (if a planning revision is still open) or, **post-seal**, refined against the real data: the methods evolve as the student learns, and the student keeps the relevant paragraph current — describing what they actually did in the Dataset Creation Paragraph (with the trail in the Log). The one thing frozen is the **Hypotheses and/or Expected Findings**.

## What "prepare" means depends on the method — the menu-shape layer

The three-phase spine below is **menu-agnostic** — access the real data, prepare it, QC it — but what "prepare" concretely produces depends on the method fixed in the Stage-14 methodology (and, per stream, whether this is the core or the extension run). The shapes mirror `operationalize-construct`'s six (same letters, so the two skills line up across planning and execution):

> **Reference — the constructed-datasets source catalog** (`design/constructed-datasets-source-catalog.md`). When a student is **building a primary dataset from scratch** (scraping, API pulls, joining sources), this catalog lists candidate sources organized by the research design each enables, with retrieval notes and licence info.
>
> **⚠️ Read it as a starting-point reference only — NOT as settled guidance.** Anything in that catalog touching ethics, sensitivity, or "who might object" is **advisory, non-decisive, and non-gating.** It is not authoritative, not a rule, and never a reason to avoid a source on its own. **The only operative question is the one `check-data-policy` answers at the time the dataset is actually being created: does this source's terms ban LLM use?** Terms and sources change; the catalog is a snapshot; the live check at creation time governs.

**A. Map construct → variable/index in existing data** — *secondary survey/administrative datasets.* Prepare: download the dataset; recode variables, handle missing data, build derived indices, restrict to the analytic sample. QC: distributions/ranges + **reproduce a published benchmark** (a known prevalence or mean). *(Fully specified below — gold exemplar #1; YRBS/MTF lineage.)*

**B. Operationalize the identification** — *quasi-experimental designs on observational data.* Prepare: assemble the identification machinery's **data inputs** — the running variable + cutoff (RD), the treatment-timing panel (DiD), the instrument, the donor pool (synthetic control) — and shape the file so the assumption checks are computable (a clean pre-period, density around the cutoff). The assumption *tests* are analysis; the *data shaped to support them* is here. QC: panel balance, pre-period coverage, cutoff-density sanity. *(Sketch; per-exemplar.)*

**C. Build a constructed quantity** — *index-building in econometrics / social epi / geo.* Prepare: access the source data, then **construct the object by the stated formula** (rank-rank slope, shift-share exposure, redlining overlay) — the derived-variable step is the heart. QC: reproduce a published value of the constructed quantity; panel balance. *(Sketch; per-exemplar.)*

**D. Measurement-model corpus** — *NLP/computational text; ML-as-measure; networks.* Prepare: **assemble the corpus** (scrape/API under the R7 reachability rules; public and secondary data need no IRB), then the **annotation/labeling pipeline + a gold-label set** (text), the feature/target table (ML), or the tie-list + structural measure (networks). QC: inter-annotator reliability / held-out accuracy of the extraction; network homophily-vs-influence sanity. *(Sketch; per-exemplar.)*

**E. Synthetic / simulation — NOT this skill.** *Synthetic AI survey participants, generative-agent simulation.* Here "sourcing data" *is generating it*, and the synthetic data is the **legitimate analyzed object** — a different integrity regime (validated by algorithmic fidelity, not "never analyzed"). Routed to **`generate-synthetic-data`** (execution). This skill never generates analyzed data. *(See the cross-skill note below.)*

**F. Structure-discovery dataset** — *causal-graphical discovery.* Prepare: assemble the **variable set + the cleaned matrix** the discovery algorithm needs (completeness/missingness suitable for the independence tests). QC: missingness pattern, sample adequacy for the chosen tests. *(Sketch; per-exemplar.)*

**Build-as-you-go flag (load-bearing, mirrors `operationalize-construct`).** Only **shape A is fully specified** here (gold exemplar #1 is observational). **Shapes B, C, D, F are sketches**, each fleshed out to full detail **when a gold exemplar in that shape is built**; **shape E is owned by `generate-synthetic-data`**. Until a shape has an exemplar, a project in it runs the spine using the shape's sketch **plus a declared deferred-scope note** in the Dataset Creation Log — so the gap is explicit, never latent.

> **Cross-skill note — the three-way synthetic-data distinction (so the regimes don't blur):**
> 1. `operationalize-construct` (planning) may build a **toy/synthetic exploration surface** — for *trying* operationalizations only; **never analyzed, never validates** a measure, never enters the paper.
> 2. **This skill** acquires **real data only** — it never generates analyzed data, and if no real dataset is reachable it routes to the research teacher or faculty mentor (gated access), never substitutes synthetic.
> 3. `generate-synthetic-data` (execution) generates synthetic/simulated data **as the legitimate studied object** — only when the **Stage-14 methodology** specifies it (shape E, or a declared cross-shape role: privacy-preserving synthesis, ML augmentation, Monte Carlo/bootstrap for inference stability, simulation for power/robustness) — validated by **algorithmic fidelity** against a real benchmark, and labeled synthetic wherever written. It may **stand in for this skill at either Data stage** — the **core** dataset at Stage 18 or the **extension** dataset at Stage 22 — for whichever stream's Stage-14 methodology names synthetic-as-object; it is never a fallback for unreachable *real* data (that stays this skill's gated-data route).

## The seven-step loop (plan → teach → execute → report)

Both Data runs (core Stage 18, extension Stage 22) run the same seven-step loop from the execution-plan model (`design/execution-plan-model-2026-07-07.md`). The gate is at the **plan** level — teach the plan, review it, confirm understanding, approve — not per line of code. AI running data-preparation code is permitted research tooling under STS (disclosed and cited); the agent writes and runs it directly. The student directs, understands, and owns the work.

**Step 1 — Plan.** The agent writes an execution plan whose purpose is to **teach**: the principles behind access/cleaning/QC for this dataset and method-shape, the ordered steps, and the **specific scripts it will write and run** (the download/fetch script, the cleaning script, the QC/benchmark script — named in the plan). The plan marks every **human-in-the-loop step** the agent cannot do itself as *"you have to do this for me — here are the steps"* — for this stage the load-bearing one is **credentialed/gated access** (ICPSR, openICPSR, UK Data Service registration and the data-use terms the student must accept; the Phase-1 gated-data moment below is exactly this element). The plan also **schedules its two Review-Agent checkpoints** — the step-3 plan review and the step-7 QC audit, each writing a **dated review `.md` to the workspace `reviews/` folder**.

**Step 2 — Teach.** The agent teaches the plan and answers questions, going deeper until the student **signals understanding**. Teaching lives up-front, in the plan — not spread across execution.

**Step 3 — Review-Agent plan review.** On the understanding-signal, the agent **composes a full handoff prompt** — pointing the **Review Agent** at the relevant docs and instructing it to write its plan review to a **dated `.md` in `reviews/`**. The student carries the prompt across; the research agent later **retrieves** the dated file, processes the changes, and **loops back through steps 1–2** (revise + re-teach) as needed.

**Step 4 — Approve.** The reviewed, revised plan is approved **jointly by the student (understanding) and the teacher**, before anything runs — a touchpoint on the process, distinct from grading the paper output.

**Step 5 — Execute (the three phases below).** The agent **writes and runs the code directly** — the download, the cleaning, the QC — following the approved plan; individual coding acts are not gated. The student performs the human-in-the-loop steps the plan assigned them (e.g., completing the credentialed download). The three phases (access → prepare → QC) are the concrete content of this step.

**Step 6 — Report.** After executing, the agent writes a **post-execution report** into the Dataset Creation Log: what it actually did, the results/diagnostics (distributions, analytic N, benchmark reproduction), and **any revision to the plan** (a script that changed, a step added, a place where the method evolved as the real data was learned — the student keeps the relevant paragraph current, which is rigorous practice).

**Step 7 — Review-Agent QC audit + spot-checks.** The agent **composes a second handoff prompt** — pointing the **Review Agent** at the **report + the executed output** and instructing it to write a **dated QC-audit `.md` to `reviews/`**; the audit flags any unexpected findings and suggests **3–5 spot checks the student runs** to catch any hallucination in the execution step. The research agent **retrieves** the dated audit file; the student runs the spot checks and reports back, and the agent prompts for this explicitly — *"tell me what you found so I can add it to your log."* **Loop-back:** an error that compromises the whole dataset → full re-run + re-report + re-audit (the stage is not approve-complete until the audit comes back clean); a smaller error → fix + a new report. Everything is logged.

## Step 5 in detail — the phases (access, prepare, QC)

### Phase 1 — Access, download, and provenance
Obtain the data and record where it came from. Many social-science datasets are **credentialed**: ICPSR (MTF, NSDUH), openICPSR/AEA replication packages, CDC YRBS (public download), UK Data Service (MCS). The skill guides the student through the actual access path — account/registration, the data-use terms they must accept, the specific file/wave to pull — and **records provenance** in `data/{core|extension}-dataset-creation-log.md` using the `regeneron-sts-reference.md` §4 fields (source, access date, version/wave, license/terms, citation). It never bypasses a login or terms agreement, and never substitutes a scraped or unofficial copy for the canonical source.

→ **Feeds the step-7 QC audit (download integrity).** The Review Agent's audit covers: right rows/waves, no skipped records, file N matches the codebook — spot checks the student runs on the executed output.

> **Scripted moment — data is gated beyond reach** *(voice-read approved 2026-07-01).* Intent: route to a human, never improvise around an access wall. *Agent:* "This dataset exists, but automatic access is blocked. Here are the specific details of what I need you to work with your research teacher or outside mentor to access directly: _____ While you're doing that, we can make a plan to clean this dataset using the information in its codebook, so we're ready the moment the real file is available."

### Phase 2 — Cleaning
Transform the raw file into the analytic dataset, **recording each decision** (not just doing it): variable recodes and how categories were formed; missing-data handling (and why — listwise, imputation, etc.); derived variables (indices, standardizations); inclusion/exclusion criteria and the resulting **analytic N** (with any subsample Ns). The cleaning log in `data/{core|extension}-dataset-creation-log.md` is the spine of the eventual Methods "sample and measures" subsection, and the major cleaning choices also go to `decisions.md` (five-field).

→ **Feeds the step-7 QC audit (cleaning).** The Review Agent's audit covers: missing-data handling, recode correctness, the load did what the log claims — spot checks the student runs on the executed output.

### Phase 3 — Quality control
Before the data is declared analysis-ready: check variable distributions and ranges for impossible values; confirm the analytic N and subsample sizes match expectations; and **benchmark against published numbers** — reproduce a known descriptive (a prevalence, a mean) from the dataset's documentation or a paper that used it, to confirm the extract is correct. Record the QC checks and their outcomes. A failed benchmark sends the agent back to Phase 2 (or Phase 1 if the wrong file was pulled). If the failure is a *measurement* problem — the variable does not capture the construct as validated — it is the loop-back call below.

Because a QC failure can mean either "wrong extract" (fix it) or "the construct isn't really in this data" (a design-level problem), the verdict is a **high-judgment moment** — so at the call the skill surfaces the research agent's **careful-tier** option (the SOUL's "How I run, and when to use a stronger model": suggest switching to the careful tier for this one decision, then dropping back). It does **not** reference the mentor's `model-escalation` meta-skill — that ships only in the project-mentor profile; the research agent's escalation surface is its own SOUL, which keeps the tiers model-agnostic. Fire it once, at the verdict — not per check.

> **Scripted moment — the QC benchmark fails** *(voice-read approved 2026-07-01; data-paragraph naming refreshed 2026-07-07 — pending re-read).* Intent: separate "wrong extract" from "construct not in the data," and slow down for the load-bearing version. *Agent:* "Our benchmark didn't reproduce — the number we should be able to hit off this file isn't coming out. That's good news in a way: QC caught something before it reached your results. Two possibilities, and they're very different. One: we pulled the wrong wave or mis-coded a step — a fix, and we go back a phase. Two: the variable just isn't carrying the construct the way the codebook promised — that's not a cleaning bug, that's a measurement problem — and your methods can keep evolving as the real data shows you what it supports; the one thing that stays frozen is your Hypotheses or Expected Findings. So let's refine now against the real data, and you describe what you actually did in your Dataset Creation Paragraph — keeping that paragraph current with the real work is exactly the rigorous thing to do. Telling those two apart is exactly the kind of call worth slowing down for, so it's worth switching to the careful tier for this one decision. Walk the failed benchmark through with me — what number did we expect, and what did we get?"

## Edge cases
- **The acquired data does not measure the construct as validated.** The Phase-3 loop-back: pre-seal → `operationalize-construct`; post-seal → refine against the real data and describe what you actually did in the Dataset Creation Paragraph (with the trail in the Log). The methods evolve as the student learns; the one thing frozen is the **Hypotheses and/or Expected Findings**, and keeping the paragraph current is rigorous practice.
- **Data exists but is gated beyond the student's access.** Record the block specifically; route to the research teacher or faculty mentor (institutional access may exist); never substitute an unofficial copy. (Phase-1 scripted moment.)
- **Cleaning would change the construct.** If the only way to make a variable usable distorts what it measures, that is an operationalization/validity problem surfacing late — loop back and report it rather than forcing the variable.
- **The Stage-14 methodology needs generated data, not sourced data.** Out of scope by design — cede to `generate-synthetic-data` (shape E and the cross-shape synthetic uses). This skill acquires real data only.
- **Student data / PII.** Out of scope by rule — this skill prepares public research datasets; student-generated data never enters shared analysis or any model.

## The data-interaction mode — read this before touching `data/`

The project's `data_interaction_mode` is set at **Stage 5**, before any agent touches a dataset, and recorded in
`data/data-validity-log.md`. **Read it at the start of every data task.** It is not advisory.

| Mode | I may receive | I may **not** receive |
|---|---|---|
| **CLEAR** | Anything, including the raw file | — |
| **ROW-RULE** | The codebook · the code I write · aggregate output · model results | **Any individual-level record** |
| **NO-AGENT** | Nothing — the student's own IRB-collected data. Local inference only. | Anything |

**ROW-RULE does not stop me executing.** I still write the script and still run it — locally, on the student's
machine, against the real file with every row present. The data never crosses the network. What is bounded is
**what comes back into my context.**

**The four leak paths — all on the return path:**

1. **Direct file reads.** No `Read`/`cat`/`head`/`less`/`grep`/`awk` on anything under `data/`. Access only via
   the analysis harness.
2. **stdout.** No `df.head()`, `print(df)`, `df.sample()`, `df.iloc[n]`. Permitted: `shape`, `dtypes`, `.info()`,
   `.describe()`, `.value_counts()`, null counts, group means, model summaries, coefficient tables, and count
   statements ("412 rows dropped for missing age").
3. **Tracebacks.** ⚠️ **The one that gets missed.** pandas dumps the offending rows in a `ParserError`; `KeyError`
   prints the row. Wrap data-touching code so errors surface as **exception type + line number + variable name** —
   never cell contents.
4. **Intermediate files.** A file written *from* `data/` inherits `data/`'s mode. No laundering rows through a
   scratch CSV.

**When I cannot diagnose without eyes on the data, I say so and tell the student exactly which rows to open and
what to look for.** The student is allowed to be the eyes. That is not a workaround — it is a person looking at
their own data.

**Why this exists.** Add Health: *"heighten external data merge risks."* ICPSR: *"not used to make connections in
the data that would increase the risk of identifying individuals."* Every archive fears the same thing — a model
used to piece identities back together out of data promised to be anonymous. **A model that never receives a
record cannot do that.** The rule is the answer to their actual concern, not a hoop.

### The policy check runs FIRST — see `check-data-policy`

**No dataset is touched until `data_interaction_mode` is set.** The check is a **precondition**, not a finding:
`check-data-policy` reads the source's terms, quotes the operative clause, sets the mode, and produces the
evidence notes the student writes up in their weekly journal. It runs at **5.3** (before any candidate dataset is
tested) and again at **18/22** (at acquisition — terms move, and the door is only fixed when the file is
downloaded). Read the mode from `data/data-validity-log.md` before doing anything else here.
